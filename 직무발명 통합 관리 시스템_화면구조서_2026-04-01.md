# 직무발명 통합 관리 시스템 화면구조서 (IA / Wire Flow)

## 1. 문서 목적

본 문서는 아시모리코리아(주) 직무발명 통합 관리 시스템의 메뉴 구조, 주요 화면 목록, 역할별 진입 경로, 핵심 화면 간 이동 흐름을 정의합니다. 기능명세서의 요구사항을 화면 단위로 재구성하여 디자인, 프론트엔드, 백엔드 구현의 공통 기준으로 사용합니다.


## 2. 역할별 대표 진입점

| 역할 | 기본 홈 | 주 사용 목적 |
|---|---|---|
| 발명자(종업원) | 내 발명 홈 | 신고서 작성, 보상/통지 확인, 동의 제출 |
| 특허관리전담부서 | 운영 대시보드 | 신고 접수, 특허/OA/비용/보상 운영 |
| 심의위원회 | 심의 대기함 | 평가표 작성, 회의록 확정 |
| 최고관리자(Admin) | 관리자 대시보드 | 정책 설정, 권한 관리, 로그/공지/동의 관리 |

## 5. 주요 화면별 핵심 구성 요소

### 5.1 SCR-102 발명 신고 작성

| 구역 | 주요 항목 |
|---|---|
| 기본 정보 | 발명명, 발명 개요, 공동 발명자, 지분율 |
| 규정 서식 입력 | 별지 제1호~4호 폼 |
| 첨부 | 양도증, 설명서, 선행기술 조사서 |
| 액션 | 임시저장, 제출, 미리보기 |

### 5.2 SCR-202 평가표 작성

| 구역 | 주요 항목 |
|---|---|
| 대상 정보 | 신고번호, 발명명, 발명자, 연결 특허 |
| 평가 요소 | 기술성, 실시가능성, 독창성, 경제적가치, 독점성, 기술의수명 |
| 계산 영역 | 요소별 선택 점수, 총점, 등급 |
| 액션 | 임시저장, 확정, 재평가 요청 |

### 5.3 SCR-402 보상 상세

| 구역 | 주요 항목 |
|---|---|
| 기본 정보 | 보상 유형, 대상 특허/신고, 정책 버전 |
| 계산 근거 | 평가 등급, 권리 유형, 산식, 순이익, 구간 |
| 배분 결과 | 총액, 발명자별 지분율, 개인별 지급 예정액 |
| 처리 이력 | 산정 시각, 승인자, 재산정 이력 |
| 액션 | 승인 요청, 반려, 재산정 요청, 통지 발송 |

### 5.4 SCR-630 실시/처분 산식 설정

| 구역 | 주요 항목 |
|---|---|
| 정책 정보 | 버전명, 시행일, 상태 |
| 산식 테이블 | 순이익 구간, 산식유형, 고정금액, 분모값, 반올림 기준 |
| 테스트 패널 | 순이익 예시 입력, 계산 결과 미리보기 |
| 액션 | 행 추가, 저장, 검증, 시뮬레이션 이동 |


## 6. 핵심 화면 흐름

### 6.1 발명 신고부터 승계 결정까지

```text
대시보드
→ 발명 신고 목록
→ 발명 신고 작성
→ 신고 상세
→ 접수/검토 큐
→ 승계 결정 상세
→ 평가 대상 목록
```

### 6.2 심의 및 특허화 흐름

```text
평가 대상 목록
→ 평가표 작성
→ 회의록 작성/승인
→ 특허 마스터 생성
→ OA 사건 상세
→ 등록 완료 처리
→ 출원/등록 통지
```

### 6.3 보상 처리 흐름

```text
보상 건 자동 생성
→ 보상 건 목록
→ 보상 상세
→ 승인 요청
→ 보상 통지 발송
→ 지급 요청/완료
→ 발명자 내 보상 현황
```

### 6.4 정책 변경 흐름

```text
정책 버전 목록
→ 정책 버전 상세
→ 등급/금액 테이블 설정
→ 평가 기준 설정
→ 실시/처분 산식 설정
→ 정책 시뮬레이션
→ 규정 변경 공지/동의
→ 적용중 전환
```

## 8. 화면 설계 시 주의사항

1. 발명자에게는 타인의 지분율, 보상액, 평가 점수가 절대 노출되지 않아야 합니다.
2. 정책 화면은 운영 입력 오류를 막기 위해 테이블 검증, 중복 구간 경고, 미리보기 기능이 우선되어야 합니다.
3. 보상 상세는 계산 투명성이 핵심이므로 최종 금액뿐 아니라 계산 경로를 단계적으로 보여줘야 합니다.
4. 모든 승인/반려/수정 제한 화면에는 변경 이력과 감사로그 링크가 있어야 합니다.
